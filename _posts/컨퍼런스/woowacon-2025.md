## 장보기∙쇼핑 가게홈 성능 최적화 고군분투기: 웹 전환 과정에서의 경험과 교훈

### 세션요약

해당 세션은 배달의민족 B마트의 웹뷰전환기에 관한 내용이었습니다.
&nbsp;

1. B마트의 화면을 고객에게 더좋은 경험을 제공, 빠른 대처(배포)를 위해 네이티브 앱에서 웹뷰로 전환함
2. 네이티브앱이 제공하는 매끄러운경험을 웹뷰로 유지해야하는 과제가 생김
3. 기존 아키텍처는 SSR(Next.js)로 이루어져있었음. 백엔드 API또한 SSR 구조를 위해 이미 설계 되었으므로 경제적으로 SSR 그대로 웹뷰 제공을 선택함
4. 배포 이후 초기 응답이 균일하지 않을까라는 기대와 달리 속도,사용성에 대한 이슈가 생김 -> 이는 곧 비즈니스에 타격을 줄 수 있는 위험성을 갖고있어서 개선 필요
5. 긴 로딩스피너 시간 이슈 + 빈화면 제공 -> 스켈레톤 제공 계획 -> SSR제공으로는 스켈레톤 제공 불가 WHY? SSR서버에 렌더링 하기위해서는 API호출 과정이 반드시 필요. 즉 API 응답지연 발생시 여전히 빈화면 제공을 하게됨 -> CSR 전환을통해 스켈레톤 시간확보 -> FCP 39%단축 성공
6. 로딩이 안끝나는 이슈 -> 보이지 않는 핸드폰 밑부분의 이미지264개를 전부 로딩을 하고있었음 -> 한번에 많은 요청 메인 스레드가 부하가 걸림 -> lazy로딩 처리 -> 또한 이미지중 1메가에 육박하는 이미지도 있었음 -> 백엔드팀과 협력해서 이미지 리사이더를 만들어 해결 LCP 10.8%, LOAD 85%개선
7. 부끄럽게도 캐시 미사용중이었음 -> IOS,안드로이드 모두 캐시 동작하도록 처리 -> 캐시정책을 바꾸는거는 다른 더미 앱의 모든 웹 페이지에 영향을 줄 수있어 전사 웹페이지 검수 조사 진행 -> FCP 32%감소, LCP 28%감소 LOAD 27%감소, AWS 비용 감소 월간비용 55.89%감소 (연3천만원의 인프라비용 절약)
8. 추가적으로 보이지 않는 영역의 자바스크립트를 서버드리븐UI방식으로 초기에 평가해야하는 스크립트 양을 줄임 -> LCP 17%감소
9. 코드스플리팅을 통한 자바스크립트 청크를쪼개기
10. 최종적으로 센트리 퍼포먼스를 모니터링 하고 특정수준 특정가드레일을 정해놓고 이 수준 이하로 나빠지면 슬랙으로 리포팅 받는 환경 구축!

### 나의 후기

슬라이드마다 FCP·LCP·Load 지표가 개선될 때마다 수치가 어떻게 달라졌는지를 명확히 보여줘서 발표에 훨씬 더 몰입할 수 있었습니다. 개선 과정을 이렇게 시각적으로 설명하면 청자 입장에서 이해와 학습 모두에 도움이 된다는 점을 다시 느꼈습니다.

아키텍처를 CSR로 바꾸기 위한 백엔드 부서와의 협업, 캐시 도입으로 인한 페이지 전수조사를 했다는 이야기를 보면서 굵직굵직한 이슈를 해결하기 위해 타 부서와의 협업이 정말 쉽지 않았겠다는 생각을 했습니다.

또한 저는 아직 웹뷰 개발을 직접 해본 경험은 없지만, 네이티브와 웹의 장단을 비교하며 아키텍처를 결정하고 측정 지표로 개선을 증명하는 접근방식은 앞으로의 개발 프로젝트에 충분히 응용할 수 있겠다는 생각이 들었습니다.

## 당연해진 디자인시스템, 그다음 이야기: AST와 MCP로 여는 미래

### 세션요약

해당 세션은 우아한 공방 팀이 AST와 MCP를 활용해 디자인시스템 일관성을 되살린 사례를 공유한 자리였습니다.
&nbsp;

1. 문제 인식: 서포트 인력은 최대 4명 수준인데 문의가 폭증해 일정 대응이 어려움
2. 가이드 반영 지연으로 서비스팀이 컴포넌트 인라인 스타일 하드코딩 삽입 → 디자인 토큰 업데이트가 전달되지 않는 악순환 발생
3. 디자인 측 진단: Figma REST API + 배치 서버로 토큰·컴포넌트 사용 현황을 JSON(S3)으로 모으고, 대시보드에서 시각화
4. Figma 플러그인 제작: 디태치된 노드 ID를 하이라이트해 디자이너가 즉시 파악 가능하도록 지원
5. 개발 측 진단: Babel Parser/Traverse 기반 Atelier Traverser 라이브러리로 AST를 순회하며 컴포넌트 임포트, 토큰 사용량, 하드코딩 스타일을 수집
6. GitLab CI 스케줄 배치로 전사 레포지토리를 자동 분석하고 결과를 S3에 저장
7. CodeMod 자동화: AST 분석 결과를 입력으로 받아 하드코딩 코드를 디자인 토큰으로 변환하고 커밋까지 생성
8. MCP 구축: 문서·피그마·스토리북·체인지로그를 Markdown/정적 파일로 변환해 리소스로 제공
9. MCP 툴: getRules, searchComponent, similarComponent 등으로 문의 응답 자동화 및 오타 교정, 레벤슈타인·야로 매칭으로 유사 컴포넌트 추천
10. MCP 프롬프트: CodeMod 실행·디자인 규칙 적용 절차를 지침 형태로 AI에게 제공해 일관된 답변/코드 생성을 유도
11. 결과: 디자인-개발 양쪽에서 디태치 구간을 정량화하고, 자동 변환 루프를 통해 우아한 공방 규칙을 다시 지킬 수 있는 환경을 마련

### 나의 후기

최근 FIGMA MCP(Cursor Talk to Figma MCP)를 테스트 해보면서 MCP로 원하는 디자인 산출물을 코드로 뽑기에는 아직이구나라는 판단을 했었던 적이 있는데 우아한 형제들은 양질의 맥락 전달을 할 수 있는 설계를 "자동화로" 구축하는구나 하면서 감탄을 하면서 들었습니다.

무엇보다 MCP 툴을 통해 스스로 커버리지를 파악하고, CodeMod 옵션까지 채워주는 데모가 강렬했습니다. AI가 규칙을 지키게 만들려면 “리소스-툴-프롬프트” 설계를 촘촘히 해야 한다는 교훈을 얻었습니다.

또한 문제해결에 있어 빅테크 기업들의 자료구조활용 방식들을 보면서 알고리즘, 자료구조의 기본기를 왜 중요시 하는지 다시한번 깨닫는 것 같습니다.
