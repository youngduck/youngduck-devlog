## 장보기∙쇼핑 가게홈 성능 최적화 고군분투기: 웹 전환 과정에서의 경험과 교훈

### 세션요약

해당 세션은 배달의민족 B마트의 웹뷰전환기에 관한 내용이었습니다.
&nbsp;

1. B마트의 화면을 고객에게 더좋은 경험을 제공, 빠른 대처(배포)를 위해 네이티브 앱에서 웹뷰로 전환함
2. 네이티브앱이 제공하는 매끄러운경험을 웹뷰로 유지해야하는 과제가 생김
3. 기존 아키텍처는 SSR(Next.js)로 이루어져있었음. 백엔드 API또한 SSR 구조를 위해 이미 설계 되었으므로 경제적으로 SSR 그대로 웹뷰 제공을 선택함
4. 배포 이후 초기 응답이 균일하지 않을까라는 기대와 달리 속도,사용성에 대한 이슈가 생김 -> 이는 곧 비즈니스에 타격을 줄 수 있는 위험성을 갖고있어서 개선 필요
5. 긴 로딩스피너 시간 이슈 + 빈화면 제공 -> 스켈레톤 제공 계획 -> SSR제공으로는 스켈레톤 제공 불가 WHY? SSR서버에 렌더링 하기위해서는 API호출 과정이 반드시 필요. 즉 API 응답지연 발생시 여전히 빈화면 제공을 하게됨 -> CSR 전환을통해 스켈레톤 시간확보 -> FCP 39%단축 성공
6. 로딩이 안끝나는 이슈 -> 보이지 않는 핸드폰 밑부분의 이미지264개를 전부 로딩을 하고있었음 -> 한번에 많은 요청 메인 스레드가 부하가 걸림 -> lazy로딩 처리 -> 또한 이미지중 1메가에 육박하는 이미지도 있었음 -> 백엔드팀과 협력해서 이미지 리사이더를 만들어 해결 LCP 10.8%, LOAD 85%개선
7. 부끄럽게도 캐시 미사용중이었음 -> IOS,안드로이드 모두 캐시 동작하도록 처리 -> 캐시정책을 바꾸는거는 다른 더미 앱의 모든 웹 페이지에 영향을 줄 수있어 전사 웹페이지 검수 조사 진행 -> FCP 32%감소, LCP 28%감소 LOAD 27%감소, AWS 비용 감소 월간비용 55.89%감소 (연3천만원의 인프라비용 절약)
8. 추가적으로 보이지 않는 영역의 자바스크립트를 서버드리븐UI방식으로 초기에 평가해야하는 스크립트 양을 줄임 -> LCP 17%감소
9. 코드스플리팅을 통한 자바스크립트 청크를쪼개기
10. 최종적으로 센트리 퍼포먼스를 모니터링 하고 특정수준 특정가드레일을 정해놓고 이 수준 이하로 나빠지면 슬랙으로 리포팅 받는 환경 구축!

### 나의 후기

우선 장표에서 FCP,LCP,Load시간 지표를 개선 하나하나 적용할때마다 바뀌어 가고있다는것을 명시적으로 표현해주어서 좀더 몰입감 있게 발표를 들었습니다. 개선과정의 발표시에 이런식으로 발표하면 청자에게 많은 도움이 되겠다는 생각을 했습니다.

어느정도 경험을 통해 알고 있는 이야기들이어서 프론트엔드 개발 공부를 헛되이 하지 않았다는 안심도 조금 했습니다.
