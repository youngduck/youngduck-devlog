> 이글은 번들 분석 도구를 써보다가 깊게 조사하게된 번들링 과 렌더링에 관한 글입니다.

## 번들 라이브러리를쓰다가..

안녕하세요! 개발자 김영덕입니다.
&nbsp;
사이드 프로젝트에 미사용 코드,라이브러리를 정리하다가 "번들 크기한번 봐볼까" 싶었습니다. 그래서 rollup-plugin-visualizer를 설치해서 번들을 시각화해봤는데요. 탭이 **Rendered**와 **Gzip** 두 개로 나뉘어져 있더라구요. 처음엔 이게 visualizer인데 왜 Gzip이 있지? 싶었고 그럼 여기서말하는 Rendered는 뭐지 했었습니다. 두 탭의 존재이유가 궁금해서 검색해보다가 제대로 딥 다이브 하게된 내용을 소개합니다.

![rendered, gzip 두가지의 탭을 제공하는 visualizer](/assets/blog/posts/devlog/devlog-6/1.png)

### Rendered탭의 의미

pnpm run build를 실행하면 dist/assets/ 폴더에 JS 파일들이 생성됩니다. 이게 **Rendered** 크기입니다. Vite가 소스코드를 트리쉐이킹, 미니파이, 청크분할 최적화를 거쳐서 만든 **최종 번들 결과물**의 크기를 의미합니다.

### Gzip탭의 의미

Rendered 파일을 Vercel이나 AWS 같은 호스팅 서버가 **자동으로 gzip 압축**해서 브라우저에 전송하는 크기를 말합니다. 이게 **Gzip** 크기입니다. (요즘은 gzip말고도 brotil등 다양한 압축방식들이 있습니다.)

### 간단한 최적화결과

![간단하게 chunk로 쪼갠 결과물](/assets/blog/posts/devlog/devlog-6/3.png)
코드스플리팅과 라이브러리 청크분리 설정을 통해 새빨갛던 bundle 데이터를 쪼개주었습니다. 적용방법은 매우 간단하여 이를 후술 하지는 않겠습니다. (~~아쉬운건 색깔을 통한 심각도를 정확히 한번에 알수가없다. 보라색은 어느정도란 말인가?~~)

## 궁금했던 점: 로컬 빌드 결과물과 Vercel에 올라간 파일은 다른건가?

번들 분석을 하다보니 이런 궁금증이 생겼습니다. 로컬에서 pnpm run build를 통해 나온 결과물은 Rendered인데, Vercel에 배포되면 Gzip으로 압축된다? 그러면 **로컬에서 빌드한 것과 실제 배포된 것이 다른 파일인건가?** 그럼 환경이 다른거네? 라고 생각했습니다.
&nbsp;
결론부터 말하면, **파일 자체는 동일합니다.** Vercel도 내부적으로 pnpm run build를 실행해서 똑같은 JS 파일을 만들어냅니다. 여기까지는 로컬이랑 완전히 같습니다.
&nbsp;
차이가 생기는 건 **전송 시점**입니다. 사용자가 사이트에 접속하면 Vercel 서버가 그 JS 파일을 gzip으로 압축해서 보내주는건데, 이건 파일을 변환하는게 아니라 **네트워크 전송 단계에서 압축하는 것**입니다. 브라우저가 받으면 자동으로 압축을 풀고, 결국 브라우저가 실행하는 건 원래의 Rendered 파일과 동일한 코드입니다.
&nbsp;
정리하면 이렇습니다.

로컬 빌드: 소스코드 → Rendered (dist/assets/\*.js) → 끝
Vercel 배포: 소스코드 → Rendered (동일) → Gzip 압축 전송 → 브라우저가 해제 → Rendered 실행

즉 **만들어지는 파일은 같고, 전송 방식만 다른 것**입니다. 로컬에서 dist 폴더 열어서 파일 크기 보면 그게 Rendered이고, 브라우저 개발자도구 Network 탭에서 Transfer Size로 보이는게 Gzip 크기입니다.

## 번들링 구조를 보다보니 — EC2와 CloudFront의 차이가 궁금해졌다

Rendered와 Gzip의 관계를 이해하고 나니, 자연스럽게 "그러면 실제 배포 환경에서는 누가 Gzip을 해주는거지?" 라는 궁금증이 이어졌습니다. 찾아보니 AWS EC2에 직접 서버를 띄우는 방식도 있고, S3 + CloudFront로 서버 없이 서빙하는 방식도 있더라구요. 이 두 구조가 어떻게 다른건지 제대로 정리해보고 싶었습니다.

### EC2 방식: 서버를 직접 띄우는 구조

EC2 하나에 프론트엔드와 백엔드를 같이 올려두는 구조가 있습니다.

```md
EC2 (하나의 리눅스 서버)
├── :3010 React → Nginx가 dist/ 정적 파일 서빙
├── :8010 Spring Boot API
└── Nginx가 리버스 프록시로 포트별 라우팅
```

EC2는 쉽게 말하면 **AWS가 제공하는 가상 컴퓨터 한 대**입니다. 서울 리전에 만들면 서울 데이터센터에 리눅스 서버가 하나 생기는거고, 거기에 Nginx 설치하고, dist 폴더 올려두고, Spring Boot도 같이 돌리는 구조입니다.
&nbsp;
여기서 중요한건 React가 EC2에서 "실행"되고 있는게 아니라는 점입니다. npm run dev로 개발서버를 띄워놓는게 아니라, npm run build로 만들어진 **dist/ 폴더의 정적 파일을 Nginx(웹서버)가 그냥 전달해주는 것**입니다. Gzip 압축도 Nginx 설정에 gzip on; 이 있으면 전송 시 해주는 겁니다.

### 그런데 Nginx는 EC2에서 기본 제공되는건가?

이것도 궁금했는데, **아닙니다.** EC2는 그냥 빈 리눅스 서버이고, Nginx는 따로 설치해야 합니다. 보통 인프라 담당자나 DevOps가 EC2 세팅할 때 sudo apt install nginx 같은 명령어로 설치하고, 포트 라우팅이나 gzip 설정 같은 것들을 nginx.conf에 직접 작성합니다.

```md
EC2 (빈 리눅스 서버)
↓ 직접 설치해야 하는 것들
├── Nginx (웹서버 + 리버스 프록시)
├── Node.js (빌드용)
├── Java (Spring Boot용)
└── 기타 필요한 런타임들
```

즉 EC2는 컴퓨터만 빌려주는거고, 그 위에 뭘 설치하고 어떻게 설정할지는 전부 개발자/인프라 담당자 몫입니다.
온프레미스도 가상화 여부만 다를 뿐, 서버 위에서 하는 작업(Nginx 설치, 설정 등)은 EC2와 동일합니다.

### S3 + CloudFront 방식: 서버 없이 서빙하는 구조

```md
GitHub Actions에서 build → dist/ 파일을 S3에 업로드 → CloudFront가 CDN으로 서빙
```

이 구조에서는 **서버가 없습니다.** S3는 파일 저장소 역할만 하고, CloudFront가 사용자에게 파일을 전달합니다. Nginx 설치? 서버 관리? 필요 없습니다.

### CloudFront(CDN)는 어떻게 동작하는가?

CloudFront는 AWS가 **전 세계에 이미 깔아놓은 엣지 서버 네트워크**입니다. 개발자가 서버를 만드는게 아니라, AWS가 이미 만들어놓은 인프라를 이용하는 겁니다.

```md
1. 개발자가 dist/ 파일을 S3에 업로드
2. CloudFront에 "이 S3를 원본으로 써라" 라고 설정
3. 유저가 접속하면:

   첫 번째 유저 (캐시 없음)
   유저 → 가까운 엣지 서버 → "나 이 파일 없는데?" → S3에서 가져옴 → 유저에게 전달 + 캐싱

   두 번째 유저 (캐시 있음)
   유저 → 가까운 엣지 서버 → "아 이거 있지" → 바로 전달 (S3 안 거침)
```

핵심은 CloudFront 자체가 컴퓨터가 아니라 **캐시 + 전달 네트워크**라는 점입니다. 연산을 하는게 아니라 파일을 캐싱해서 빠르게 뿌려주는 역할만 합니다. Gzip/Brotli 압축도 CloudFront에서 해주는데, 기본값은 꺼져있고 **"Compress objects automatically" 옵션을 켜야** 동작합니다.

### 전체 플로우를 한눈에 보면

배포 방식에 따라 Rendered 이후의 경로만 달라지고, 빌드까지는 전부 동일합니다.

```
                     ┌─────────────────────────┐
                     │   React 소스코드 (TSX)    │
                     │   + node_modules         │
                     └────────────┬────────────┘
                                  │
                           npm run build
                       (트리쉐이킹 + 미니파이)
                                  │
                                  ▼
                     ┌─────────────────────────┐
                     │  ✅ Rendered             │
                     │  dist/assets/*.js        │
                     │  (여기까지는 전부 동일)    │
                     └────────────┬────────────┘
                                  │
           ┌──────────────────────┼──────────────────────┐
           ▼                      ▼                      ▼
  ┌────────────────┐   ┌────────────────┐   ┌────────────────┐
  │   AWS EC2      │   │ S3+CloudFront  │   │    Vercel      │
  │                │   │                │   │                │
  │ CodeDeploy로   │   │ dist/를 S3에   │   │ git push하면   │
  │ dist/ 배치     │   │ 직접 업로드     │   │ 자동 빌드+배포  │
  └───────┬────────┘   └───────┬────────┘   └───────┬────────┘
          │                    │                     │
          ▼                    ▼                     ▼
  ┌────────────────┐   ┌────────────────┐   ┌────────────────┐
  │  Nginx         │   │  CloudFront    │   │  Vercel Edge   │
  │  (직접 설치)    │   │  (AWS CDN)     │   │  (자체 CDN)    │
  │                │   │                │   │                │
  │ nginx.conf에   │   │ 설정만 하면     │   │ 설정 필요 없음  │
  │ gzip on; 설정  │   │ 자동 압축       │   │ 자동 압축       │
  └───────┬────────┘   └───────┬────────┘   └───────┬────────┘
          │                    │                     │
          └────────────────────┼─────────────────────┘
                               │
                               ▼
                  ┌─────────────────────────┐
                  │  ✅ Gzip 압축 전송       │
                  │  Content-Encoding: gzip  │
                  └────────────┬────────────┘
                               │
                               ▼
                  ┌─────────────────────────┐
                  │  🌐 브라우저             │
                  │  gzip 자동 해제          │
                  │  → 원본 JS 파싱 → DOM 생성 │
                  │  → 화면에 그림             │
                  └─────────────────────────┘
```

차이는 **누가 Gzip 압축을 해주느냐** 뿐입니다. EC2는 Nginx를 직접 설치하고 설정해야 하고, CloudFront와 Vercel은 자동으로 해줍니다. 그리고 브라우저가 최종적으로 실행하는 코드는 어떤 방식이든 **전부 같은 Rendered 파일**입니다.

### 두 구조를 비교하면

|           | EC2 + Nginx              | S3 + CloudFront    |
| --------- | ------------------------ | ------------------ |
| 서버      | 직접 관리 (리눅스)       | 없음 (서버리스)    |
| 파일 서빙 | Nginx                    | CloudFront (CDN)   |
| Gzip 압축 | Nginx 설정 필요          | Compress 옵션 필요 |
| 속도      | EC2 위치 기준            | 전 세계 엣지 서버  |
| 비용      | EC2 24시간 과금          | 트래픽만큼만 과금  |
| 관리 부담 | 높음 (OS, Nginx 설정 등) | 낮음               |

프론트엔드를 서빙하는 용도라면 S3 + CloudFront가 확실히 표준인것 같고 회사 상황에 맞게 잘선택하면 될 것 같습니다.

## 실제로 Gzip 전송을 확인해봐야지

여기까지 이해하고 나니, 실제 서비스에서 정말 Gzip으로 전송되는지 직접 확인해보고 싶었습니다. 브라우저 개발자도구 Network 탭에서 확인할 수 있습니다.
&nbsp;
네이버 사이트에서 아무 요청이나 클릭해서 Response Headers를 봤더니

![네이버 응답 헤더에서 Content-Encoding: gzip 확인](/assets/blog/posts/devlog/devlog-6/2.png)

**Content-Encoding: gzip** 이 보였습니다. 요게 "이 응답은 gzip으로 압축되어 전송되었다"는 의미입니다.
&nbsp;
이게 가능한 이유는 HTTP 프로토콜에 이런 약속이 있기 때문입니다:

```
브라우저 → 요청 헤더: "Accept-Encoding: gzip" (나 gzip 풀 수 있어)
서버    → 응답 헤더: "Content-Encoding: gzip" (이거 gzip으로 보낼게)
브라우저 → 자동으로 압축 해제 후 실행
```

즉 **브라우저는 네트워크로는 Gzip 상태로 받고, 자동으로 해제한 뒤 JS를 파싱해서 실행**하는 겁니다. 개발자도구에서 코드를 클릭해서 보면 이미 해제된 원본 상태입니다.

## 총정리

### 1. 번들링 → 전송 → 렌더링

```
React 소스코드 (~300kB라 가정)
    ↓  [번들링] Vite가 트리쉐이킹 + 미니파이 + 청크 분할
Rendered (~120kB) ← 빌드 결과물, dist/assets/*.js
    ↓  배포 시점에 웹서버/CDN에 세팅해놓음
    ↓
    ↓  [전송] 유저가 요청할 때마다 웹서버(Nginx) 또는 CDN(CloudFront, Vercel)이 gzip/brotli 압축
Gzip (~30kB) ← 네트워크로 실제 전송되는 크기
    ↓  [렌더링] 브라우저가 gzip 해제 → 다시 120kB → JS 파싱 → React 실행 → DOM 생성 → 화면에 그림
```

- **번들링**: 소스코드 → Rendered (개발자가 최적화할 수 있는 영역)
- **전송**: Rendered → Gzip (웹서버 또는 CDN이 유저 요청 시마다 담당)
- **렌더링**: Gzip 해제 → 브라우저가 JS를 파싱하고 실행 → DOM 생성 → 화면에 그리는 영역

### 2. 청크 분할과 유저 플로우

번들링 시 청크 분할(Code Splitting)을 하면 하나의 큰 파일이 아니라 페이지별로 쪼개집니다. React.lazy를 사용하면 해당 페이지에 접근할 때 비로소 그 청크만 다운로드하는 지연 로딩(Lazy Loading)이 가능합니다.

```
[청크 분할 안 했을 때]
유저가 로그인 페이지 접속
  → index.js (500kB) 전체를 gzip 압축 → ~150kB 전송 → 느림

[청크 분할 했을 때]
유저가 로그인 페이지 접속
  → index.js (100kB) + login.js (50kB)만 gzip 압축 → ~45kB 전송 → 빠름

유저가 대시보드로 이동
  → dashboard.js (80kB)를 그때서야 gzip 압축 → ~24kB 전송

유저가 어드민으로 이동
  → admin.js (120kB)를 그때서야 gzip 압축 → ~36kB 전송
```

총 다운로드량은 비슷할 수 있지만, **한 번에 gzip 압축해서 보내는 양 자체가 줄어드니까** 초기 로딩 속도가 크게 빨라집니다.

### 3. 배포 방식별 차이

배포 방식에 따라 전송을 담당하는 주체만 달라집니다.

- **EC2**: Nginx를 직접 설치하고 gzip 설정도 직접 해야 함
- **S3 + CloudFront**: "Compress objects automatically" 옵션을 켜면 CDN이 gzip/brotli 압축 + 전 세계 엣지 서버에서 캐싱
- **Vercel**: 빌드부터 CDN 전송까지 전부 자동
